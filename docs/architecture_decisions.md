## ADR-001：统计方法而非 ML 用于异常检测

**日期**: 2026-05-20  
**状态**: 已采纳  
**决策者**: WANG Chuncheng

### 背景

供应链异常监控需要在每日 18 万行订单数据中识别异常，并将结果推送给非技术业务人员（运营经理、财务分析师）。检测结果需要可解释——业务人员必须理解"为什么这笔订单是异常"。

### 选项

| 选项 | 描述 |
|------|------|
| A. Isolation Forest | 无监督 ML，输出异常分数 |
| B. LSTM Autoencoder | 深度学习方法，学习正常模式后检测偏离 |
| C. 纯统计方法（Z-Score + IQR + 移动平均） | 经典统计，可解释 |

### 决策

**选择 C：纯统计方法。**

### 理由

1. **可解释性是硬性要求**：Z-Score 告诉运营"这笔订单利润偏离均值 2.5 个标准差"，Isolation Forest 只能说"异常分数 0.73"。前者可以直接转化为处置建议，后者是黑盒。
2. **零训练成本**：18 万行数据跑 Isolation Forest 需要训练 + 超参数调优，Z-Score 是 O(n) 的单次计算。
3. **阈值透明**：z=2.5 的阈值可以追溯到具体的分位数——"这个阈值覆盖了约 2.5% 的订单"。ML 模型的阈值通常缺乏这种直观对应。
4. **供应链数据分布特征**：DataCo 数据集的利润分布是右偏的（median $31.5 vs mean $22.0），Z-Score 对正态假设不完美，但 IQR 对偏态分布鲁棒，两者组合互补覆盖。
5. **业务规则兜底**：有 6 条硬规则可以直接拦截"绝对异常"（如利润 < -$200、延迟 > 3 天），这些不需要统计模型。

### 后果

- **正面**：业务人员理解并信任检测结果，飞书推送后可直接决策。
- **负面**：无法发现高维特征空间中的复杂模式（多因素微弱共现），这方面 ML 方法有优势。
- **缓解**：异常模式聚类（`pattern_clusterer.py`）部分弥补了这一不足——通过规则聚类识别"延迟+亏损复合"等模式。

## ADR-002：阈值 2.5 而非教科书默认的 3.0

**日期**: 2026-05-20  
**状态**: 已采纳  
**决策者**: WANG Chuncheng

### 背景

Z-Score 异常检测的教科书默认阈值是 3.0（对应正态分布约 0.27% 的尾部）。但 DataCo 数据集的实际分布并非正态，直接套用教科书值会严重漏检。

### 决策

**选择 z=2.5**，基于 DataCo 数据集的实际分位数推算。

### 分析过程

```
Benefit per order: mean=22.0, std=104.4
- z=3.0 下界：22.0 - 3×104.4 = -291.3 → 只能捕获 < 0.5% 的订单
- z=2.5 下界：22.0 - 2.5×104.4 = -239.1 → 覆盖约 2.5% 的订单（6,041 条亏损记录）
- 5th%ile = -$139, 1st%ile = -$416
- z=2.5 刚好落在 5th%ile 和 1st%ile 之间
```

### 理由

1. 从数据推导，不盲从教科书。config.yaml 里每个阈值都有注释说明数据来源。
2. 在 Precision 和 Recall 之间找到了平衡：z=2.5 检出 ~6K 条异常，不会淹没飞书推送，也不会漏掉大额亏损。
3. 其他阈值同理——IQR 乘数 k=2.0（基于实际 IQR 分布），移动平均 deviation_factor=2.0。

### 后果

- **正面**：阈值有据可查，面试或 Code Review 时可以清楚解释"为什么是 2.5"。
- 换数据源后需要重新推算阈值——这不是一次性的配置，而是需要随数据更新持续维护的分析。

## ADR-003：Shipping Mode 不作为预测特征

**日期**: 2026-05-20  
**状态**: 已采纳  
**决策者**: WANG Chuncheng

### 背景

在 EDA 阶段发现：First Class 运输方式的延迟率高达 95.3%，Standard Class 仅 38.1%。直观上会认为"First Class 导致延迟"，应该用 Shipping Mode 作为异常检测的输入特征。

### 决策

**Shipping Mode 不作为预测特征。**

### 理由

1. **数据泄漏**：First Class 高延迟率的原因是**物流系统会对已延迟的订单"升级"运输方式试图补救**。Shipping Mode 是延迟的**结果**（post-hoc 变量），而非延迟的原因。
2. 如果建模时使用了 Shipping Mode，模型会学到"First Class → 高延迟"这个虚假关联，在生产环境中会错误地对所有 First Class 订单标记为异常。
3. 这是教科书级别的数据泄漏案例：在预测时能看到"补救措施的痕迹"，但补救措施发生在异常之后。

### 分析证据

- First Class 仅占 288 笔订单（0.16%），且几乎全部处于 Late delivery 状态
- Standard Class 占 108,245 笔（60%），延迟率仅 38%
- 时序分析显示：订单先被标记为 Late delivery，随后 Shipping Mode 才被升级

### 后果

- **正面**：避免了生产环境中的系统性误报。
- 这个发现被记录在 `notes.md` 中，成为项目的数据质量警觉性案例。

## ADR-004：不引入向量数据库（轻量级 RAG 替代方案）

**日期**: 2026-05-20  
**状态**: 已采纳  
**决策者**: WANG Chuncheng

### 背景

归因 Agent 需要在分析异常时引用供应链 SOP 知识（10 条标准处置流程）。典型的 RAG 方案是：向量化 SOP → 存入向量数据库 → 根据异常指标做语义检索 → 嵌入 prompt。

### 选项

| 选项 | 描述 |
|------|------|
| A. ChromaDB / FAISS 向量检索 | 完整的 RAG pipeline |
| B. 关键词匹配 + 全量注入 | 简单规则匹配 |

### 决策

**选择 B：关键词匹配 + 全量注入。**

### 理由

1. **知识库极小**：10 条 SOP 总共约 2,000 字，全量注入 prompt 仅增加 ~500 tokens，远低于 DeepSeek 的 64K 上下文窗口。
2. **关键词匹配精度足够**：SOP 条目标题本身包含明确的领域术语（"延迟飙升"、"大额亏损"、"利润率异常"），用异常指标名直接映射到 SOP 编号即可。
3. **避免过度工程化**：向量数据库需要额外的持久化、索引维护、embedding 调用。对于 10 条 SOP，引入这些基础设施的复杂度远超收益。
4. **可解释性**：关键词匹配规则是透明的——"Benefit per order 异常 → SOP-002"这条规则一眼就能验证正确性。向量检索的匹配结果是不可解释的。

### 后果

- **正面**：项目保持轻量，无外部向量数据库依赖，新人 Onboarding 只需 `pip install -r requirements.txt`。
- **负面**：如果 SOP 增长到数百条，全量注入不再可行，届时需要迁移到向量检索。当前架构预留了 `_match_sops()` 接口，替换实现不影响上层调用。

## ADR-005：DeepSeek 而非 GLM/Claude 用于归因分析

**日期**: 2026-05-20  
**状态**: 已采纳  
**决策者**: WANG Chuncheng

### 背景

归因 Agent 需要调用 LLM 进行根因分析，每批次 15-30 条。选择 LLM 的核心考量：成本、结构化 JSON 输出可靠性、API 兼容性。

### 选项

| 选项 | 单条成本 | JSON 可靠性 | API 兼容性 |
|------|----------|-------------|-----------|
| DeepSeek V3/V4 | ~$0.001 | 高 | OpenAI SDK |
| GLM-4 | ~$0.01 | 中 | 独立 SDK |
| Claude (Anthropic) | ~$0.03 | 极高 | 独立 SDK |
| GPT-4o (OpenAI) | ~$0.05 | 高 | OpenAI SDK |

### 决策

**选择 DeepSeek V4 Flash（日常）/ DeepSeek V3（备选）。**

### 理由

1. **成本优势**：每日 15 条归因 × 30 天 = 450 条/月，DeepSeek 月成本约 $0.45，Claude 月成本约 $13.5。30 倍差距。
2. **OpenAI SDK 兼容**：`base_url` 改为 `api.deepseek.com` 即可切换，无需改代码。换模型只需改 `config.yaml` 中的 `model` 字段。
3. **结构化输出稳定性**：经过评测（见 `eval/run_eval.py`），V4 Flash 在 `max_tokens=4096` 下 JSON 解析成功率达 100%（`max_tokens=2048` 时为 64%）。V3 为 95%。
4. **表达多样性**：V4 Flash 比 V3 表现出更好的句式多样性（DeepSeek 评委评分 4/5 vs 2/5），归因结果读起来更自然。
5. **跨模型评委**：评测时用 MiniMax 当评委评 DeepSeek 的归因质量，避免自评作弊。

### 后果

- **正面**：月成本可忽略不计，适合个人项目/找实习场景。
- **负面**：DeepSeek 的 API 稳定性偶尔波动（偶发 500 错误），重试逻辑（最多 3 次，指数退避）已覆盖。
- **切换路径**：如果未来需要更高的归因质量，`config.yaml` 改一行即可切换到 Claude/GPT-4o——所有 LLM 调用统一走 `AttributionAgent.call_llm()`。

## ADR-006：规则聚类而非 ML 聚类用于异常模式识别

**日期**: 2026-05-21  
**状态**: 已采纳  
**决策者**: WANG Chuncheng

### 背景

单条归因报告的信息密度有限——"订单 #33 延迟+亏损"和"订单 #64 延迟+亏损"本质上是同一类问题。需要将异常归类为"模式"，让决策者看到全局而非个案。

### 选项

| 选项 | 描述 |
|------|------|
| A. DBSCAN / Jaccard 距离聚类 | 基于特征距离的无监督聚类 |
| B. 规则聚类 | 人工定义模式（延迟+亏损、品类集中、区域集中） |

### 决策

**选择 B：规则聚类。**

### 理由

1. **业务语义 > 数学距离**：DBSCAN 会把"订单 #33 延迟+亏损"和"订单 #35 延迟"聚在一起（因为都有延迟特征），但业务上这是两类问题——前者是复合异常（需财务+物流联合处置），后者是纯物流问题。规则聚类尊重业务语义。
2. **解释性**：规则聚类的结果可以直接变成飞书卡片里的文案——"269 个订单呈现延迟+亏损复合特征"。DBSCAN 的"Cluster #0: 1,292 points, centroid: [...]" 业务人员完全看不懂。
3. **离散特征不适合距离计算**：品类名、区域名、运输方式全是离散类别变量，Jaccard 相似度只有 0/1，区分度不足。
4. **规则可随时调整**：想增加一个新模式（如"大额订单 + 新客户异常"），只需在 `pattern_clusterer.py` 加一个聚类函数 + 在 `config.yaml` 加一行配置，不需要重新训练。

### 后果

- **正面**：6 个模式直接呈现在飞书日报卡片中，业务人员优先关注的"延迟+亏损复合"模式排在第一位。
- **负面**：无法发现规则未定义的模式（未知模式）。如果未来数据中出现规则外的新型异常模式，需要通过人工分析发现后补充规则。
- **缓解**：保留 `config.yaml` 中的 `method: "rule_based"` 选项，为未来实现 `method: "distance_based"` 留好扩展点。
